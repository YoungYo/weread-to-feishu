# weread-to-feishu v1 改造计划

## 目标

把当前“仅支持覆盖写入飞书文档”的插件，改造成一个更合理的 v1：

- 免费主入口：**导出本地 Markdown**
- 保留现有能力：**同步到飞书文档（覆盖写入）**
- 在界面与文案上明确区分两种能力的定位与风险

---

## 改造原则

1. **先降门槛，再谈扩展**
   - 本地 Markdown 导出不依赖飞书配置，应成为默认低门槛入口。

2. **尽量少动现有稳定链路**
   - 飞书同步能力先保留，不做大重构，只做入口整理与风险提示。

3. **让产品表达更清晰**
   - 用户一眼能看懂：
     - 这个插件能干什么
     - 哪个能力适合先试
     - 飞书同步会覆盖原文档

4. **控制本轮改动范围**
   - 本轮不做：Notion、flomo、增量同步、账号体系、服务端。

## 暂不处理的问题

以下问题确认存在，但不在本轮解决范围内：

- 飞书同步仍是“先删后写”的高风险模型
- 多本书仍未拆为“每本书一个飞书文档”
- 微信读书网页接口变动风险仍在
- Markdown -> Feishu Block 的转换能力仍较简化
- 没有增量同步、失败恢复、断点续传

---

## 完成本轮后的预期结果

完成后，插件会变成：

1. **普通用户可直接试用**
   - 登录微信读书网页版后，无需飞书配置即可导出 Markdown。

2. **飞书同步能力继续保留**
   - 面向有明确飞书沉淀需求的用户。

3. **产品结构更合理**
   - 免费入口更自然
   - 飞书能力更像高级用法

## 下一阶段架构升级点（先写入计划，暂不改代码）

以下升级点已确认有价值，但当前**只纳入计划，不在本轮立即实施**。

### 1. 拆薄 `List.svelte`
现状问题：
- 当前 `List.svelte` 同时承担了 UI 展示、状态管理、流程编排、导出触发等职责，已经偏重。

计划方向：
- 将“本地导出流程”和“飞书同步流程”从页面组件中拆出去
- 页面层只保留：
  - 选择书籍
  - 收集配置
  - 展示状态
  - 触发动作
- 业务流程下沉到独立 action / service 模块

目标收益：
- 页面更容易维护
- 后续新增导出目标时，页面无需继续膨胀

### 2. 给导出目标建立统一抽象
现状问题：
- 当前虽然已经有“本地 Markdown 导出”和“飞书同步”两个出口，但代码层还没有统一抽象。

计划方向：
- 引入统一的导出目标接口，例如：
  - local markdown exporter
  - feishu exporter
  - future: notion exporter / flomo exporter
- 统一输入与执行方式，减少分支散落在页面层

目标收益：
- 新增导出目标时更顺
- 更容易做免费版 / 付费版能力切换

### 3. 从 Markdown 中间层进一步升级为“标准笔记模型层”
现状问题：
- 当前 `weread.ts` 直接把微信读书数据整理成 Markdown，这在当前阶段足够快，但会限制后续更复杂的导出能力。

计划方向：
- 增加一层标准化的 note model，例如统一表达：
  - 书籍元信息
  - 章节结构
  - 划线
  - 评论
  - 阅读进度
- Markdown 变成其中一个 renderer，而不是唯一中间表达

目标收益：
- 更适合支持：
  - 高质量飞书 block 映射
  - Notion / flomo 等不同目标
  - 更强的排版与样式控制

### 4. 拆分飞书适配层职责
现状问题：
- 当前 `feishu.ts` 同时负责鉴权、文档地址解析、限流重试、Markdown 转 block、删除和写入文档，职责偏重。

计划方向：
- 后续可拆成：
  - feishu auth
  - feishu document client
  - feishu markdown/block renderer
- 把“接口调用”和“内容转换”分开

目标收益：
- 更利于排查问题
- 更方便后续迭代飞书写入策略（比如从覆盖写入升级到更安全的方式）

### 5. 明确插件的目标架构演进路线
建议后续逐步收敛为：

```text
WXT / Chrome Extension
  ↓
Popup UI（轻展示、轻编排）
  ↓
Use Cases / Actions（导出、本地下载、飞书同步）
  ↓
Domain Model（标准笔记模型）
  ↓
Adapters
  - WeRead Source Adapter
  - Local Markdown Exporter
  - Feishu Exporter
  - Future: Notion / flomo
```

### 6. 下一阶段建议顺序
先后顺序建议如下：

1. 先输出“下一阶段重构方案文档”（不改代码）
2. 再拆 `List.svelte` 的流程编排
3. 再抽导出目标统一接口
4. 再决定是否引入标准笔记模型层

这样能避免一次性重构过大，保证每一步都可验收。
